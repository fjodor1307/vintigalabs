## 2026-05-07 вҖ” Responsive shell вҖ” sidebar collapses to drawer below `md`

The dashboard shell didn't honour the design rule (DESIGN-RULES.md: "Sidebar hidden below `md` (768px), mobile overlay on toggle"). Every layout was hard-coding `Navbar device="desktop"` and the inline sidebar stayed visible at every viewport, squeezing content below tablet width.

Now lives in the DS so every prototype gets responsive chrome for free:

- New hook `@ds/shared/useResponsiveSidebar` вҖ” single source of truth for `collapsed` (md+ icon-only) + `mobileOpen` (sub-md drawer). `onMenuToggle` does the right thing per viewport via `matchMedia`. Pair with `<Navbar device="responsive" onMenuToggle={onMenuToggle} />` and `<AppSidebar collapsed mobileOpen onMobileClose />`.
- `AppSidebar` rebuilt as two render paths: desktop wrapper `hidden md:flex` (inline, takes layout space) + mobile fixed overlay `md:hidden` with backdrop dismiss. Both share an internal `AppSidebarInner` so nav stays in one place. `onItemClick` fires on every nav item in the mobile drawer so tapping a destination closes the drawer too.
- All seven layouts swept to use the hook + `device="responsive"`:
  - `customers/CustomerViewLayout` + `customers/CustomersScreen`
  - `clubs/ClubsLayout` + `clubs/ClubViewLayout` + `clubs/ClubEditorLayout`
  - `products/Shell` + `products/ProductLayout`
  - `onboarding/DashboardScreen` (uses bare `Sidebar` so the same wrapper pattern is inlined; nav extracted into a local `DashboardSidebar` component)

Verified end-to-end in preview at 375Г—812 (mobile) and 1280Г—800 (desktop): inline sidebar disappears below md, hamburger appears, drawer slides in, backdrop click + nav-item tap both dismiss, viewport-resize back to md+ restores the inline sidebar.

## 2026-05-07 вҖ” PageTemplate: Tailwind UI 2:1 grid for main + rail

`PageTemplate` switched from "flex with a fixed 360-px aside" to a CSS-grid two-column shell:

```
grid grid-cols-1 gap-vintiga-lg lg:grid-cols-3
  в”ңв”Җ <main>  lg:col-span-2  (2/3 width)
  в””в”Җ <aside> lg:col-span-1  (1/3 width)
```

Why: align with the Tailwind UI page template the team is standardising on, so every "main + rail" page (Customer Overview, Product detail, New Club editor, etc.) shares one shape regardless of viewport width. The previous fixed 360-px aside drifted in proportion as the viewport grew; the grid keeps the 2:1 ratio at every `lg+` breakpoint.

Side-effects:

- The 84-px top inset on the rail is gone вҖ” the first rail card now aligns with the breadcrumb at the top of the main column (matches the Customer Overview Figma, where Notes sits at breadcrumb level).
- Outer padding moves up one wrapper: `p-vintiga-xl` is on the grid container, not on `<main>`. Same total padding, just a different element owns it.
- Below `lg`, the rail still stacks under the main column (now via `grid-cols-1`, no longer a flex-direction switch).
- No API change. All three consuming prototypes (Customers, Products, Clubs) pick this up automatically.

## 2026-05-04 вҖ” KpiCard: three variants from Figma (132:6062)

Brought `@ds/shared/KpiCard` in line with the Figma source of truth. The component now covers all three documented variants on a single API:

1. **Plain** вҖ” `label` + `value` + `icon`
2. **Goal / progress** вҖ” adds `goalLabel` + `progressPercent` (renders the progress strip below the value)
3. **Status pill** вҖ” adds `status?: ReactNode` slot (e.g. `<Tag variant="outline">Awaiting first sale</Tag>`) inline with the value. Combine with goal/progress freely вҖ” the "AOV $0 / Awaiting first order / No goal set 0%" empty state from Figma renders correctly.

Other tweaks:
- Icon container resized 40 вҶ’ **36 px** (matches Figma) and forces 20 px child SVGs via `[&>svg]:w-5 [&>svg]:h-5`. Existing usages don't need to pass `w-* h-*` on the icon.
- Progress section now renders whenever `goalLabel` is set, even without an explicit `progressPercent` (defaults to 0% вҖ” supports the "No goal set" state).

Style guide entry refreshed at `#/web/design-system вҶ’ Data Display вҶ’ KPI Card` with a row per variant.
